# Архитектура ETL / Парсера

Назначение: периодически извлекать цену из одного или нескольких источников (магазинов) и превращать “сырье” в нормализованные данные `PriceSnapshot`, которые сохраняются в PostgreSQL (через backend/API или напрямую по согласованию).

## Принципиальные идеи MVP
- **Адаптер = источник**: один адаптер отвечает за извлечение цены для одного `parser_type` (`Source.parser_type`).
- **Нормализация перед записью**: до сохранения приводим цену/валюту/временную метку к единому формату.
- **Ошибки тоже являются данными**: при ошибках сохраняем `PriceSnapshot.status=error` (с `error_message`), чтобы видеть качество адаптера.
- **Идемпотентность**: ретраи не должны плодить дубль‑снимки (обычно через уникальность по `(product_id, fetched_at)` или дедупликацию на уровне ETL).

## Пайплайн (логическая схема)
1. `Scheduler` выбирает, какие `Product` нужно обновить (по расписанию).
2. `ETL Runner` создаёт “job” на каждый `Product`.
3. `Adapter` получает вход:
   - `source.base_url`,
   - `product.url`,
   - настройки (таймауты, headers, задержки).
4. `Adapter`:
   - делает HTTP запрос,
   - получает HTML/JSON,
   - извлекает цену/валюту (и опционально наличие/наличие в корзине).
5. `Normalizer` приводит результат к контракту `PriceSnapshot`:
   - `price` -> `DECIMAL(12,2)` (строка/число внутри кода),
   - `currency` -> ISO 4217 код (`RUB`),
   - `fetched_at` -> timestamptz в UTC.
6. `Persister` сохраняет:
   - через POST `/api/v1/price-snapshots/internal`, или
   - напрямую в БД (только если согласовано с Лёшей).
7. `Logger/Monitor` пишет логи и агрегирует метрики (успех/ошибка/время).

## Интерфейсы адаптеров (контракт внутри проекта)

### `SourceAdapter` (минимальный)
- `parser_type` (строка, совпадает с `Source.parser_type`)
- `extract_price(product_url) -> ExtractionResult`

### `ExtractionResult`
- `price` (число или строка, приводимая к decimal)
- `currency` (строка, например `RUB`)
- `provider` (строка: имя адаптера/магазина)
- `raw_data` (опционально: snippet/JSON для дебага)
- `error` (опционально: структурированное сообщение)

## Ретраи и обработка ошибок
Классы ошибок (в MVP):
- **Timeout / connection error**: ретрай с backoff.
- **404/Not found**: считать `status=error` (или “success=false”), но фиксировать `error_message` (например `product_not_found`).
- **Price not found / selector changed**: фиксировать `status=error` + указать, что именно не удалось извлечь.
- **Rate limit / blocked**: ретраи позже (backoff), плюс уменьшение конкурентности.

Рекомендованные настройки:
- `REQUEST_TIMEOUT_SECONDS` (например 20)
- `MAX_RETRIES` (например 3)
- `RETRY_BACKOFF_SECONDS` (например 2, 4, 8)
- `FETCH_DELAY_SECONDS` (задержка между запросами одного адаптера)
- `MAX_CONCURRENCY` (ограничение параллелизма)

## “Вежливый” парсинг
- Учитывать `robots.txt` (минимально: не нарушать явно запрещённые пути).
- Добавлять корректный `User-Agent`.
- Использовать заголовки и кэширование, если источник поддерживает.
- Добавлять задержки между запросами.

## Конфигурация
Переменные окружения (примерно):
- `DATABASE_URL` (если ETL пишет в БД напрямую)
- `API_BASE_URL` + `ETL_API_KEY` (если ETL пишет через backend)
- `SOURCES_TO_SYNC` (на MVP: ограничить список)
- `SCHEDULE_CRON` (если APScheduler/cron в контейнере)

## Точка расширения (добавление нового магазина)
При добавлении нового источника:
- добавить запись `Source` в БД (или seed-данные),
- реализовать новый адаптер под соответствующий `parser_type`,
- добавить параметры/селекторы и обновить описания в документации (в этом же файле, секция “Adapter notes” — по согласованию).

## Adapter notes (шаблон для новых адаптеров)
Используйте единый шаблон, чтобы остальные могли быстро подключить/проверить источник:

- **parser_type**: например `ozon_html`
- **Источник**: base URL, примеры страниц товара
- **Где извлекаем цену**:
  - HTML селектор / JSON-LD поле / мета-теги
- **Валюта**: фиксированная (`RUB`) или извлекаемая
- **Стратегия ошибок**:
  - если цена не найдена — как отличаем “товар не найден” от “селектор сломался”
- **Ограничения**:
  - rate limit, возможная капча, рекомендуемые задержки
- **Как тестировать**:
  - 2–3 URL для smoke-теста
  - ожидаемый формат результата `ExtractionResult`
