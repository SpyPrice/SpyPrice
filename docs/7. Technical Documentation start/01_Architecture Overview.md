# Архитектурный обзор

Цель MVP — показать сквозной сценарий:
1) пользователь добавляет товар (URL),
2) система хранит связь `User → Product → Source`,
3) ETL периодически извлекает цены и пишет `PriceSnapshot`,
4) фронтенд читает историю/метрики и визуализирует динамику.

## Компоненты
- **Frontend (SPA)**: авторизация, управление товарами, графики/таблицы.
- **Backend API (FastAPI)**:
  - auth (регистрация/логин),
  - CRUD для `Product`,
  - чтение истории/метрик,
  - приём данных ETL (служебный эндпоинт или internal-процедуры).
- **ETL / Parser**:
  - scheduler запускает задания,
  - адаптеры источников (по одному на магазин),
  - извлечение → нормализация → сохранение.
- **PostgreSQL**: хранит схему домена (User/Product/Source/PriceSnapshot/Alert*).

## Взаимодействия (логические)
- Frontend ↔ Backend API: только через REST API.
- ETL ↔ Backend API (предпочтительно для консистентности):
  - ETL отправляет `PriceSnapshot` через internal/endpoints,
  - backend обеспечивает валидацию и целостность.
- Backend ↔ PostgreSQL: чтение/запись сущностей и агрегатов.
