# Правила оформления коммитов

Данный документ описывает стандарты оформления коммитов для всех участников проекта. Соблюдение этих правил обеспечивает единообразие истории изменений и упрощает работу с `git log`, `git blame` и автоматической генерацией `CHANGELOG`.

## 1. Структура коммита

Каждый коммит должен состоять из трех частей:

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 1.1. Type (Обязательно)

Определяет тип изменения. Используйте только один из следующих вариантов:

| Type       | Описание                                                                                                 |
| ---------- | -------------------------------------------------------------------------------------------------------- |
| `feat`     | Новая функциональность (добавлена возможность, новая фича).                                              |
| `fix`      | Исправление ошибки (бага).                                                                               |
| `docs`     | Изменения только в документации (README, JSDoc, комментарии).                                            |
| `style`    | Изменения, не влияющие на логику кода (форматирование, пробелы, точки с запятой, правки линтера).        |
| `refactor` | Рефакторинг кода (не новая функциональность и не исправление бага).                                      |
| `perf`     | Изменения, направленные на улучшение производительности.                                                 |
| `test`     | Добавление или исправление тестов.                                                                       |
| `build`    | Изменения, влияющие на систему сборки или внешние зависимости (npm, gradle, webpack).                    |
| `ci`       | Изменения в конфигурации CI/CD (GitHub Actions, GitLab CI, Jenkins).                                     |
| `chore`    | Вспомогательные задачи, не связанные с кодом продукта (обновление зависимостей, настройка инструментов). |
| `revert`   | Отмена предыдущего коммита.                                                                              |

### 1.2. Scope (Опционально)

Область кода, которую затронуло изменение. Указывается в круглых скобках.

**Примеры:** `auth`, `api`, `ui`, `parser`, `config`, `deps`, `frontend`, `backend`.

### 1.3. Subject (Обязательно)

Краткое, емкое описание изменения.

- Используйте **повелительное наклонение** (императив): "add" вместо "added" или "adds".
- Не ставьте точку в конце заголовка.
- Ограничивайте длину **50 символами**.

### 1.4. Body (Опционально)

Подробное описание _почему_ было сделано изменение и _как_ оно работает.

- Используется, если изменение требует контекста.
- Длина строки ограничена **72 символами** для читаемости в терминале.
- Здесь можно описывать мотивацию и отличие от предыдущего поведения.

### 1.5. Footer (Опционально)

Используется для:

- **Закрытия Issues**: `Closes #123`, `Fixes #456`, `Resolves #789`.
- **Указания на breaking changes**: `BREAKING CHANGE: описание изменений, нарушающих обратную совместимость`.

---

## 2. Примеры

### ✅ Хорошие примеры

```
feat(auth): add two-factor authentication support
```

```
fix(api): handle null response in user profile endpoint

Previously, if the API returned null for the profile, the app would crash.
Now we return an empty object and log a warning.

Closes #42
```

```
docs(readme): update installation guide for Node.js 20
```

```
refactor(parser): simplify regex logic for performance

Extracted duplicate patterns into reusable constants.
This reduces CPU usage by ~15% during large file parsing.
```

### ❌ Плохие примеры

```
fix bug
```

```
Обновил файлы и пофиксил баги
```

```
Update code
```

```
feat: сделал
```

## 3. Рекомендации по работе с ветками

1.  **Ветка разработки:** `dev` или `main`.
2.  **Ветки задач:** Называйте по шаблону `<type>/<short-description>`.
    - _Пример:_ `feat/add-login-page`, `fix/button-styling`, `docs/update-api-spec`.
3.  **Pull Requests:** Название PR должно отражать суть изменений (желательно повторять `subject` основного коммита). В описание PR добавляйте `body` и `footer`.

---

## 4. Breaking Changes

Если ваше изменение ломает обратную совместимость (меняет API, требует миграции данных, удаляет функционал):

1.  Укажите `BREAKING CHANGE:` в футере коммита.
2.  Добавьте восклицательный знак после `type`/`scope`.

**Пример:**

```
feat(api)!: remove deprecated v1 endpoints

The legacy API endpoints have been removed. Please migrate to v2.

BREAKING CHANGE: /api/v1/users is no longer available. Use /api/v2/users instead.
```

---

_Правила вступают в силу с текущей даты. Нарушение этих правил является поводом для запроса изменений (request changes) при ревью Pull Request._
