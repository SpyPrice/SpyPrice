# Роли команды и зоны ответственности

Проект: **сервис отслеживания и анализа изменения цен в онлайн‑магазинах** (MVP).

## Соответствие методичке (ТЗ) и фактический состав

В [`ТЗ.md`](../../ТЗ.md) для команды из **4 человек** рекомендованы роли: Team Lead / PM, Backend, Data/ETL, Frontend **или** QA. У нас **5 участников**: роль **DevOps** вынесена отдельно (Рома) — это напрямую покрывает требования кейса к **Git, контейнеризации, деплою и базовому CI/CD**. Приёмочное тестирование и проверка сценариев MVP распределяются: **Егор** (регресс по UI и интеграции), **Фёдор** (приёмка по продукту и сценариям показов), при необходимости подключаются **Лёша/Эдик** для API и сборщика.

| Роль в ТЗ | У нас |
|-----------|--------|
| Team Lead / Project Manager | Фёдор |
| Backend-разработчик | Лёша |
| Data/ETL | Эдик |
| Frontend / QA | Егор (frontend + тестирование интерфейса); QA по продукту — см. выше |
| DevOps (в компетенциях кейса) | Рома |

## Фёдор — Team Lead / Project Manager + Дизайнер
- **Ответственность**: процесс и синхронизация команды, планирование спринтов, бэклог, подготовка **четырёх показов** и финальной отчётности; UI/UX, wireframes, прототипы; фасилитация Agile/Scrum (бэклог, приоритеты, демо).
- **Артефакты**: user stories + acceptance criteria, макеты/прототип (Figma), презентации к показам, структура документации в `docs/`, контроль полноты **корневого README** перед защитой.

## Егор — Frontend-разработчик (и приёмка UI)
- **Ответственность**: SPA-интерфейс, визуализация истории цен (графики/таблицы), интеграция с API, адаптивность и UX-состояния; smoke- и регресс-проверки ключевых пользовательских сценариев перед показами.
- **Артефакты**: код фронтенда, прототип интерфейса (в коде), согласованные с бэкендом типы/DTO (см. `Contracts.md`).

## Лёша — Backend-разработчик (Python)
- **Ответственность**: архитектура сервера, API, бизнес‑логика, модель данных, авторизация (JWT), миграции, документация OpenAPI; разграничение доступа к данным по пользователям.
- **Артефакты**: схема БД (ER), спецификация API, каркас приложения, миграции, демо‑данные.

## Рома — DevOps / Инфраструктура
- **Ответственность**: воспроизводимое окружение (Docker/Compose), CI (линтеры, сборка, проверки на PR), публичный **деплой стенда** для демонстраций; секреты только в переменных окружения, не в репозитории.
- **Артефакты**: Dockerfile(ы), `docker-compose` (dev/prod), GitHub Actions, инструкции по локальному запуску и выкладке (дублируются в корневом `README` по требованию ТЗ).

## Эдик — Data/ETL инженер
- **Ответственность**: источники данных, адаптеры парсинга, планировщик/очередь сбора (по мере внедрения), нормализация и сохранение `PriceSnapshot`, устойчивость к ошибкам; соблюдение **robots.txt**, rate limit и этичного объёма запросов.
- **Артефакты**: код ETL/парсера, описание поддерживаемых источников и ограничений (в README и в `Parser Architecture.md`), протокол сохранения срезов (через API или согласованная запись в БД).
