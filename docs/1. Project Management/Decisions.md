# Лог архитектурных решений (ADR)

Формат: дата → решение → краткая мотивация/последствия.

## 2026-03-18 — Базовый стек для MVP
**Решение**:
- **Backend**: Python + **FastAPI**
- **DB**: **PostgreSQL**
- **Frontend**: **React** (желательно с TypeScript)
- **Инфраструктура**: **Docker** (+ docker-compose для dev/prod)

**Почему**:
- FastAPI даёт быстрый старт, асинхронность и удобный OpenAPI `/docs`.
- PostgreSQL подходит для связей и целостности (User ↔ Product ↔ PriceSnapshot), индексов по времени.
- React — стандартный выбор для SPA и удобен для графиков (Recharts/Chart.js).
- Docker упрощает воспроизводимую разработку, демо‑стенд и CI.

**Следствия**:
- Нужно согласовать модели/контракты между фронтом и бэком (Pydantic ↔ TS интерфейсы).
- Для планировщика на старте можно взять простой вариант (APScheduler), с возможностью перейти на Celery при росте.

## 2026-03-18 — Сбор данных: этика и ограничения (из ТЗ)

**Решение**: реализовывать сбор только в рамках учебного проекта: приоритет **публичные API** и **ограниченный** набор сайтов; соблюдать **robots.txt**, разумные задержки между запросами (**rate limiting** с нашей стороны), корректный `User-Agent`; не создавать чрезмерную нагрузку на магазины.

**Почему**: это явное требование задания и необходимое условие защиты (вопросы по ограничениям и правовым аспектам входят в критерии).

**Следствия**: список поддерживаемых источников и правила вежливого сбора документируются в README и в [`Parser Architecture.md`](../7.%20Technical%20Documentation/Parser%20Architecture.md).
