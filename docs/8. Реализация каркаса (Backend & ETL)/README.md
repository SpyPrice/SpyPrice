# Реализация каркаса: Backend и ETL

Этот раздел описывает **фактический код** в репозитории: что уже сделано, как это запускается и какие есть ограничения. Концептуальные контракты и планы по API продолжают жить в [`7. Technical Documentation/`](../7.%20Technical%20Documentation/); при расхождении с кодом **источник истины для поведения сервиса — код и этот файл**, пока документация не синхронизирована вручную.

## Назначение каркаса

Реализован **качественный технический каркас** под зону ответственности **Лёши (backend)** и **Эдика (ETL)**:

- HTTP API для пользователя: регистрация, вход, список источников, CRUD отслеживаемых товаров, история цен, агрегированные метрики.
- **Internal ingest** для записи срезов цен с ETL по секретному ключу.
- Прототип ETL с адаптерами **DNS** и **Ozon** (HTML + эвристика), отправка срезов в backend.
- Запуск через **Docker Compose**: PostgreSQL + API; контейнер ETL запускается вручную (profile `manual`).

Аналитика «топ изменений» и экспорт CSV **в коде не реализованы** (по договорённости ограничились ядром + internal ingest).

## Стек

| Компонент | Технология |
|-----------|------------|
| API | Python 3.12, FastAPI, Uvicorn |
| ORM | SQLAlchemy 2.x |
| БД | PostgreSQL 16 (официальный образ) |
| Доступ к БД | драйвер `psycopg` (пакет `psycopg[binary]`) |
| Аутентификация пользователя | JWT (HS256), заголовок `Authorization: Bearer <token>` |
| Защита ingest | заголовок `X-ETL-API-Key` |
| ETL | httpx, BeautifulSoup4 |

**Миграции Alembic** в каркасе не подключены: схема создаётся через `Base.metadata.create_all()` при старте приложения. Для продакшена позже стоит перейти на Alembic.

## Архитектура (логическая)

```text
┌─────────────┐     JWT      ┌──────────────────┐     SQL      ┌────────────┐
│   Client    │ ───────────► │  Backend (API)   │ ───────────► │ PostgreSQL │
│  (browser)  │              │  /api/v1/...     │              │            │
└─────────────┘              └────────▲─────────┘              └────────────┘
                                      │
                           X-ETL-API-Key
                                      │
                               ┌──────┴──────┐
                               │ ETL runner  │
                               │  (manual)   │
                               └─────────────┘
```

- Пользовательские запросы идут в API с JWT.
- ETL ходит только на `POST /api/v1/price-snapshots/internal` с ключом (не с JWT пользователя).

## Модель данных (реализованная)

Соответствие таблицам SQLAlchemy в `backend/app/models.py`:

| Таблица | Назначение |
|---------|------------|
| `users` | Пользователь: email, хеш пароля, даты, soft-delete флаг |
| `sources` | Магазин/источник: имя, базовый URL, `parser_type` (уникальный ключ для выбора адаптера в ETL) |
| `products` | Отслеживаемый товар пользователя: `user_id`, `source_id`, URL, имя, теги (JSON-массив), soft-delete |
| `price_snapshots` | Срез цены: `product_id`, цена, валюта, `fetched_at` (UTC), `status`, текст ошибки, `availability`, `raw_data` |

**Ограничения:**

- Уникальность пары `(product_id, fetched_at)` — защита от дублей при повторной отправке того же времени; при конфликте insert API возвращает `ok: true` без второй записи (идемпотентность на уровне БД).
- `status` у снимка — строка (`success` / `error`); в коде не вынесено в PostgreSQL ENUM.
- `availability` — строка; прототип ETL всегда шлёт `unknown` (извлечение наличия можно добавить в адаптеры позже).

## Сиды источников

При старте backend в таблицу `sources` добавляются записи, если их ещё нет (по уникальному `parser_type`):

| name | base_url | parser_type |
|------|----------|-------------|
| DNS | https://www.dns-shop.ru | `dns_html` |
| Ozon | https://www.ozon.ru | `ozon_html` |

UUID источника для создания товара берётся из `GET /api/v1/sources`.

## API: реализованные маршруты

Базовый префикс: **`/api/v1`**. Интерактивная документация: **Swagger UI** по адресу `/docs` (когда поднят backend).

### Аутентификация

| Метод | Путь | Описание |
|--------|------|----------|
| POST | `/auth/register` | Регистрация: `email`, `password` (мин. 8 символов). Ответ: `user_id`, `email`. |
| POST | `/auth/login` | Вход: те же поля. Ответ: `access_token`, `token_type`, `expires_in` (секунды). |
| GET | `/auth/me` | Текущий пользователь (нужен Bearer-токен). |

### Источники и товары

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/sources` | Список источников (без авторизации). |
| GET | `/products` | Список товаров текущего пользователя. Query: `limit`, `offset`, `source_id`, повторяющийся `tags`. |
| POST | `/products` | Создание: `url`, `name`, `source_id`, `tags`. |
| PATCH | `/products/{product_id}` | Частичное обновление: `name`, `tags`. |
| DELETE | `/products/{product_id}` | Soft-delete (`is_deleted = true`). |

**Фильтр `tags`:** передаётся как несколько параметров, например `?tags=a&tags=b`. В SQLAlchemy используется `Product.tags.contains(...)` — для JSONB-массива тегов это означает, что в сохранённом массиве должны присутствовать переданные значения (уточняйте поведение при интеграции фронта, при необходимости замените на пересечение множеств).

### История и метрики

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/products/{product_id}/history` | История срезов. Query: `from`, `to`, `limit`, `offset`. |
| GET | `/products/{product_id}/metrics` | Метрики по **успешным** срезам (`status=success`). Query: либо `from`+`to`, либо **`preset=7d` или `preset=30d`** (окно от «сейчас» назад). Считаются min, max, avg, delta и delta_percent между первой и последней точкой в выборке. |

### Internal (ETL)

| Метод | Путь | Заголовки | Описание |
|--------|------|-----------|----------|
| POST | `/price-snapshots/internal` | `X-ETL-API-Key: <секрет>` | Тело: `product_id`, `price`, `currency`, `fetched_at` (ISO 8601), `status`, опционально `error_message`, `availability`, `raw_data`. Товар не должен быть удалён (`is_deleted=false`). |

### Ошибки

В каркасе для ошибок используется стандартное поведение FastAPI: чаще всего JSON вида `{"detail": "текст"}` или `{"detail": [...]}` для валидации. Отдельный единый формат `ErrorResponse` из проектной документации **пока не внедрён** во все handler’ы.

## Безопасность (что уже есть)

1. **Пароли** — хранятся как bcrypt-хеш (`passlib`).
2. **JWT** — подпись HS256, секрет `JWT_SECRET`; время жизни токена — поле `access_token_ttl_minutes` в `app/config.py` (по умолчанию 60 минут). В окружении см. имя переменной для pydantic-settings: обычно `ACCESS_TOKEN_TTL_MINUTES`.
3. **Доступ к данным** — операции с товаром и историей проверяют `user_id` владельца.
4. **Ingest** — без корректного `X-ETL-API-Key` запрос отклоняется (403).
5. **CORS** — список origin’ов из переменной `CORS_ORIGINS` (через запятую).

Секреты не коммитить: используйте `.env` (файл в `.gitignore`), образцы — в `.env.example`.

## Переменные окружения

### Docker Compose (`docker-compose.yml`)

Переменные ниже пробрасываются из shell или из файла `.env` в корне проекта (Compose читает `.env` автоматически).

| Переменная | Где используется | Смысл |
|------------|------------------|--------|
| `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` | сервис `db`, строка подключения backend | учётные данные БД |
| `JWT_SECRET` | backend | секрет подписи JWT |
| `ETL_API_KEY` | backend + etl | общий ключ для internal ingest |
| `CORS_ORIGINS` | backend | разрешённые origin’ы для браузера |
| `DEMO_ITEMS` | etl | строка заданий для ручного прогона (см. ниже) |

Внутри контейнера backend строка БД собирается как:

`postgresql+psycopg://USER:PASSWORD@db:5432/DB`

### Локальный запуск backend без Compose (опционально)

Файл `backend/app/config.py` читает `.env` из **текущей рабочей директории** процесса. Удобно запускать из каталога `backend/` с собственным `.env` или экспортом переменных. Переменная **`DATABASE_URL`** должна указывать на доступный PostgreSQL.

## ETL: как устроен и как запускать

Файл: `etl/app.py`.

1. Читает список заданий из **`DEMO_ITEMS`**: формат  
   `product_id|parser_type|url;product_id|parser_type|url;...`  
   где `parser_type` — `dns_html` или `ozon_html` (должен совпасть с полем источника у товара в БД).

2. Для каждой записи: HTTP GET страницы (с `User-Agent`, таймаутом, редиректами), затем **грубое** извлечение цены регулярным выражением по тексту страницы (символ ₽/р). Это **прототип**: вёрстка магазинов меняется, возможны капча, 403 и неверная цена.

3. Успех → `POST .../price-snapshots/internal` с `status=success`. Не найдена цена или исключение → запись с `status=error` и сообщением (чтобы история фиксировала попытки).

4. Между запросами — пауза `FETCH_DELAY_SECONDS` (вежливость к сайту).

Переменные ETL:

| Переменная | По умолчанию | Смысл |
|------------|----------------|--------|
| `API_BASE_URL` | `http://backend:8000` | Базовый URL API (в Compose — имя сервиса `backend`) |
| `ETL_API_KEY` | `change-me-etl-key` | Должен совпадать с ключом в backend |
| `REQUEST_TIMEOUT_SECONDS` | 20 | Таймаут HTTP |
| `MAX_RETRIES` | 3 | Повторы при сетевых ошибках (экспоненциальная пауза) |
| `FETCH_DELAY_SECONDS` | 1 | Пауза после обработки одной записи |

Запуск одноразового контейнера:

```bash
docker compose --profile manual run --rm etl
```

`DEMO_ITEMS` задайте в `.env` в корне репозитория или передайте в командной строке окружения.

## Правовые и этические ограничения

Сбор с DNS и Ozon в учебных целях должен быть **умеренным**: не ддосить, соблюдать разумные паузы, учитывать правила сайтов и роботов. Текущий код не парсит `robots.txt` автоматически — это намеренный компромисс каркаса; для отчётности и защиты команда должна описывать политику и ограничения частоты (см. также [`ТЗ.md`](../../ТЗ.md)).

## Известные ограничения и следующие шаги

- Нет refresh-токенов; один access JWT до истечения срока.
- Нет отдельной роли администратора.
- Нет планировщика периодического сбора — только ручной запуск ETL.
- Парсинг цен эвристический; для стабильности нужны отдельные селекторы/API по каждому магазину.
- Желательно: Alembic, единый формат ошибок API, тесты, CI.
